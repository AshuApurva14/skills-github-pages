Secure CI/CD: Integrating GitHub Actions with JFrog via OIDC 

In modern DevOps, storing long-lived secrets like passwords or API keys in GitHub is a security risk. By using OpenID Connect (OIDC), you can establish a trust relationship between GitHub Actions and the JFrog Platform, enabling a "tokenless" and more secure workflow. 

Why Use OIDC with JFrog?

No Stored Secrets: Eliminate the need to store JFrog credentials as GitHub Secrets.
Short-Lived Tokens: Tokens are generated on-the-fly for each job and expire automatically, reducing the attack surface.
Fine-Grained Access: Use Identity Mappings to ensure only specific repositories or branches can access your artifacts. 

How It Works

Request: The GitHub Actions workflow requests an OIDC token from GitHub.
Verify: JFrog validates the token against GitHub’s public keys to verify the workflow's identity.
Access: Once verified, JFrog issues a short-lived access token to the workflow for downloading or publishing artifacts. 

Quick Setup Steps

Configure JFrog: Navigate to Administration > General Management > Integrations in the JFrog UI. Create a new OpenID Connect integration for GitHub.
Create Identity Mappings: Define which GitHub repositories or environments are allowed to connect and map them to specific JFrog permissions.
Update GitHub Workflow: Add the required permission to your .yml file to allow it to request the OIDC token:
```yaml
yaml
permissions:
  id-token: write
  contents: read
```
Authenticate: Use the JFrog Setup Action or JFrog CLI to authenticate using the configured OIDC provider name. 

By adopting OIDC, you transition from managing static credentials to a dynamic, identity-based security model that scales with your development team.

Below is a sample GitHub Actions workflow for a Docker build and push using JFrog OIDC. 

GitHub Actions Workflow: Docker Build & Push via OIDC 

This workflow uses the jfrog/setup-jfrog-cli action to authenticate without stored secrets, then leverages the generated short-lived token to log in to the Docker registry. 

```yaml
yaml
name: "JFrog Docker OIDC Build"
on:
  push:
    branches: [ "main" ]

permissions:
  id-token: write # Required for requesting the OIDC JWT
  contents: read  # Required for actions/checkout

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    env:
      # Replace with your actual JFrog URL and repository name
      JF_URL: https://mycompany.jfrog.io
      DOCKER_REPO: "my-docker-local"
      IMAGE_NAME: "my-app"

    steps:
      - name: Checkout Code
        uses: actions/checkout@v4

      - name: Setup JFrog CLI
        uses: jfrog/setup-jfrog-cli@v4
        id: setup-cli
        with:
          # This name must match the 'Provider Name' configured in JFrog
          oidc-provider-name: "jfrog-github-oidc"
        env:
          JF_URL: ${{ env.JF_URL }}

      - name: Authenticate Docker
        uses: docker/login-action@v3
        with:
          registry: ${{ env.JF_URL }}
          # The setup-cli action automatically outputs the OIDC user and token
          username: ${{ steps.setup-cli.outputs.oidc-user }}
          password: ${{ steps.setup-cli.outputs.oidc-token }}

      - name: Docker Build and Push
        run: |
          # Define the full image tag
          IMAGE_TAG=${{ env.JF_URL }}/${{ env.DOCKER_REPO }}/${{ env.IMAGE_NAME }}:${{ github.run_number }}
          
          # Build using standard Docker or jf docker build to collect build-info
          jf docker build -t $IMAGE_TAG .
          jf docker push $IMAGE_TAG

```
